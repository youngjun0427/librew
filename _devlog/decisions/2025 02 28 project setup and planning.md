# 바이브 코딩으로 사이드 프로젝트 세팅하기 — AI 기획부터 컨텍스트 관리까지

## 시작하게 된 배경

스타트업 3년. 기술적으로 빠르게 치고 나가는 환경이었지만 돌아보면 구멍이 있었다. 테스트 코드를 진지하게 작성한 적이 없고, CI/CD는 누군가 세팅해둔 걸 그냥 썼고, 성능 최적화는 "느리면 고친다" 수준이었다. 대기업 공고를 보면 이런 것들이 자격요건으로 올라와 있다. 우대도 아니고 자격요건으로.

그래서 이번 사이드 프로젝트는 처음부터 이 구멍들을 메우는 방향으로 설계했다. 주제는 커피 핸드드립 앱. 취미로 하던 거라 도메인 이해가 있고, 기능이 명확하게 정의되는 도메인이라 설계 연습하기 좋았다.

---

## AI로 기획하는 방식

기획에 AI를 쓴다는 게 처음엔 어색했다. 근데 실제로 써보니 **내 생각을 빠르게 구조화하는 도구**로서 효과가 있었다. 중요한 건 AI가 결정하는 게 아니라 내가 방향을 잡고 AI가 검증하고 구체화하는 흐름이다.

실제로 유효했던 방식은 두 가지다.

첫째, **기능을 던지고 분해시키는 것**이다. "레시피 저장"이라고 하면 AI가 베이스 레시피 / 장비별 파생 / 원두별 파생을 분리해서 생각하자고 제안했다. 혼자 생각할 때는 뭉뚱그려 넘어갔을 부분을 쪼개주는 역할이다.

둘째, **MVP 범위를 조율하는 것**이다. "한 달 MVP 기준으로 잘라줘"라고 하면 AI가 기능 간 의존 관계를 기준으로 우선순위를 제안한다. 여기서 내가 동의하는 것과 동의 안 하는 것을 구분해서 조율한다. AI 의견을 그대로 따르는 게 아니라 내 판단의 근거를 빠르게 검증하는 용도다.

## 기획 과정에서 실제로 고민한 것들

**웹이냐 앱이냐**

웹 개발자라 React 기반으로 짜고 싶었는데 앱스토어 등록도 필요했다. PWA를 쓰면 앱스토어 등록이 안 되고, React Native를 처음부터 배우기엔 비용이 컸다. Expo의 React Native Web이 절충점이었다. 코드 한 벌로 iOS / Android / 웹을 커버하고, JSX와 훅 문법은 그대로 쓴다. 스타일링만 NativeWind로 바꾸면 된다.

전략은 웹 기준으로 기능을 먼저 완성하고 앱 UI를 다듬는 순서. 브라우저에서 빠르게 확인하면서 개발하고 완성도가 올라오면 플랫폼별로 조정하는 흐름이다.

**AI 기능 범위**

처음엔 AI 기능을 배제하려 했다. 비용과 복잡도 때문에. 근데 Gemini 무료 티어가 하루 1500회 요청이고, 개인 앱에서 실제 호출량은 훨씬 적다. 그래서 관능 평가 후 다음 추출 개선 제안 하나만 넣기로 했다. API 키 노출 방지를 위해 Firebase Functions 경량 프록시를 두는 구조로 설계했다.

**스택 선정 기준**

구직 공고 10개 정도를 분석해서 공통으로 나오는 키워드를 추렸다. TypeScript, 상태관리, 테스트 코드, CI/CD, 성능 최적화, 그리고 최근에는 AI 활용 경험이 자격요건으로 올라오고 있다. 이 프로젝트 하나로 전부 커버하는 게 목표였다.

Zustand를 선택한 이유는 Firebase 실시간 구독과 조합할 때 보일러플레이트가 적고 구조가 단순하기 때문이다. Redux를 쓸 수도 있었지만 Firebase SDK가 이미 비동기 처리를 담당하고 있어서 Zustand의 단순한 스토어 구조가 오히려 맞는다고 판단했다. 이런 선택의 근거가 면접에서 쓸 수 있는 이야기가 된다.

## AI 컨텍스트 관리 전략

바이브 코딩의 실질적인 문제는 세션이 끊길 때마다 컨텍스트가 날아간다는 거다. 매 세션마다 프로젝트 배경을 다시 설명하는 건 비효율적이고, 설명이 부족하면 AI가 엉뚱한 방향으로 코드를 짠다.

해결 방향은 두 가지였다.

하나는 CLAUDE.md를 프로젝트 루트에 두는 것. Claude Code나 Cursor는 프로젝트 열 때 이 파일을 자동으로 읽는다. 근데 여기에 모든 걸 때려박으면 컨텍스트 윈도우를 너무 많이 잡아먹는다. 그래서 CLAUDE.md는 핵심만 30줄 이내로 유지하고, 상세 문서는 분리했다.

```
CLAUDE.md                    ← 스택, 플랫폼, 문서 위치만
_docs/context/
├── architecture.md          ← Firestore 구조, 화면 설계
├── conventions.md           ← 코드 규칙, 네이밍, 폴더 구조
├── ai-strategy.md           ← Gemini 프롬프트 전략, 프록시 구조
└── devlog-rules.md          ← devlog 작성 규칙
```

세션을 시작할 때 작업 성격에 따라 필요한 문서만 주입한다.

```
기능 개발:    CLAUDE.md + architecture.md 읽고 원두 CRUD 작업할게
버그 디버깅:  CLAUDE.md + conventions.md 읽고 이 에러 봐줘
AI 기능:      CLAUDE.md + ai-strategy.md 읽고 프롬프트 튜닝해줘
```

전체 컨텍스트를 항상 주입하는 게 아니라 그 세션에서 필요한 것만 올리는 방식이다. 이렇게 하면 AI 응답 품질도 올라간다. 관련 없는 정보가 많을수록 노이즈가 생긴다.

## 개발하면서 블로그도 쓰는 구조

사이드 프로젝트를 하면서 블로그도 병행하고 싶었는데 가장 큰 문제는 귀찮음이다. 개발하고 나서 "나중에 정리해야지" 하면 디테일이 다 날아간다.

해결책은 **기록 비용을 최소화하는 구조**를 만드는 거다. `_devlog/` 폴더를 프로젝트 안에 두고, CLAUDE.md에 규칙을 넣었다.

> 에러 해결, 새 패턴 적용, 설계 결정이 있으면 devlog 파일을 생성해라.

개발하다가 뭔가 해결되는 순간 AI가 알아서 `_devlog/troubleshooting/2025-03-01-nativewind-setup.md` 를 만들어준다. 그 자리에서 완성된 글을 쓰는 게 아니라 상황, 에러 메시지, 시도한 것들, 해결 방법을 로그 수준으로 남기는 거다.

나중에 블로그 포스트로 만들고 싶으면 그 파일을 던지면서 "블로그 초안 만들어줘"라고 하면 된다. 글쓰기 비용이 메모 → AI 초안 → 다듬기로 줄어든다.

블로그 글감 목록도 미리 정해뒀다. 초기 세팅, Firestore 구조 설계, 추출 도우미 타이머 구현, Firebase Functions 프록시, Lighthouse 최적화 등. 개발 순서와 블로그 주제가 1:1로 매핑되는 구조라 "이 기능 개발하면 이 글 쓴다"는 흐름이 자연스럽게 잡힌다.
