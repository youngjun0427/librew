## 점검 순서

### 1. 커밋 히스토리 파악
```bash
git log --oneline          # 전체 흐름 파악
git log --oneline -10      # 최근 10개
git diff HEAD~5 --stat     # 최근 5커밋 변경 파일 목록
```
어떤 기능이 추가됐는지, 커밋 단위가 적절한지 확인.

### 2. 코드 회고 (Code Retrospective)
최근 변경된 파일들을 읽고 아래 항목 체크:

**품질 기준 준수 여부** (`quality-standards.md` 기준):
- [ ] 비동기 스크린에 isLoading / error 처리 있는가
- [ ] 폼 버튼 isSubmitting disabled 처리 있는가
- [ ] `as Type` 무조건 캐스팅 없는가
- [ ] TypeScript 에러 없는가 (`tsc --noEmit`)

**아키텍처**:
- [ ] 비즈니스 로직이 훅에 있는가 (스크린에 직접 없는가)
- [ ] 컴포넌트 크기 적절한가 (500줄 기준)
- [ ] 스토어 역할이 명확한가 (도메인 데이터 캐시)

**발견한 문제** → 즉시 수정하거나 TODO 이슈로 남기기

### 3. 개발 회고 (Dev Retrospective)
코드 너머의 개발 과정을 돌아본다:

**잘한 것**:
- 어떤 설계 결정이 좋았는가?
- 어떤 패턴을 잘 적용했는가?

**배울 점**:
- 이번에 처음 쓴 패턴/API는 무엇인가?
- "왜 이렇게 짰는가?" 설명할 수 있는가?
- 다음에 같은 상황이면 어떻게 할 것인가?

**놓친 것**:
- 기술 부채로 남긴 것은?
- 급하게 처리해서 나중에 손봐야 할 곳은?

### 4. 학습 포인트 정리
면접에서 말할 수 있는 수준으로 정리:

```
이번에 배운 것:
- [기술/패턴] — [한 줄 설명] — [왜 이렇게 했는가]

면접 답변 연습:
- "이 프로젝트에서 가장 어려웠던 점은?" → [답변]
- "왜 이 기술 스택을 골랐어요?" → [답변]
- "Zustand와 Firestore를 어떻게 연결했어요?" → [답변]
```

### 5. 다음 우선순위 제안
남은 작업 중 포트폴리오 임팩트 순서로 정렬해서 제안.

## 회고 출력 형식

Claude는 아래 형식으로 결과를 출력한다:

```
## 🔍 코드 회고
### 잘 된 부분
### 개선 필요 부분 (+ 수정 제안)

## 📚 개발 회고
### 이번에 배운 것
### 놓친 것 / 기술 부채

## 💬 면접 답변 연습
[질문] → [답변]

## 🎯 다음 작업 우선순위
1.
2.
3.
```

## 중간점검 주기 권장
- 기능 단위 완성마다 (5~10커밋)
- 막히는 느낌이 들 때
- "지금 내가 제대로 가고 있나?" 의심이 들 때
